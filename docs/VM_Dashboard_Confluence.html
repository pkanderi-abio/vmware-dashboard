VMware Dashboard – Technical Documentation

Infrastructure Monitoring Dashboard for VMware vCenter Environments

Version: 2.6
Last Updated: April 2026
Owner: Cloud Infrastructure Engineering (CIE)
Status: Production
Overview
The VMware Dashboard is an internal web application that provides real-time visibility into VMware vCenter environments. It aggregates data from multiple vCenter instances, enriches it with PuppetDB configuration management data, and maintains a historical CMDB of VM inventory. Engineers use it to monitor ESXi hosts, virtual machines, datastores, networks, and snapshots from a single unified interface.

1. Architecture Overview

The VMware Dashboard follows a two-tier web application pattern: a Python FastAPI backend that collects and serves data, and a React single-page application that consumes the API. Both run on the same physical server as separate systemd services.

Key Design Decisions

DecisionRationale
In-process persistent cache (JSON on disk) vCenter API calls are slow (10–30 s). A 30-minute in-memory cache with disk persistence means fast page loads and resilience across service restarts.
Background refresh thread Data collection runs in a daemon thread so API requests always return immediately from cache.
Historical CMDB in JSON VM inventory is persisted locally to track decommissioned machines over time without requiring a database.
pyVmomi + PropertyCollector Uses VMware's native Python SDK with batch PropertyCollector calls for high-performance data retrieval across large environments.
Vite proxy for API calls Frontend calls /api/* on the same origin; Vite proxies to the backend. No CORS header manipulation needed in production.

4. Backend – Python / FastAPI

4.1 Module Descriptions

main.py – Application entry point and API router

The core of the backend. Version 2.6. Responsibilities:

  • FastAPI application setup with CORS middleware
  • pyVmomi session management (VCenterPyVmomi class with auto-reconnect)
  • In-memory + on-disk PersistentCache class (30-minute TTL)
  • Background data refresh thread (parallel collection from all vCenters)
  • Credential loading/saving to vcenter_credentials.json
  • All REST API endpoint definitions
  • Startup: connects to all saved vCenters, seeds health checker, starts refresh
  • Shutdown: graceful disconnect from all pyVmomi sessions

vcenter_fast.py – Parallel vCenter data collector

Provides collect_vcenter_data_parallel() which uses Python's ThreadPoolExecutor to concurrently collect data from all configured vCenters. Uses pyVmomi's PropertyCollector for efficient batch retrieval of VM, host, datastore, and network properties. This avoids the N+1 query problem common with traversal-based collection.

cmdb_history.py – Historical VM inventory (CMDB)

Implements the HistoricalCMDB class. Key capabilities:

  • Persists complete VM inventory to ~/.vmware-dashboard-cache/cmdb_history.json
  • Tracks VM lifecycle: first_seen, last_seen, status (active / decommissioned / unreachable)
  • Detects decommissioned VMs when they disappear from a connected vCenter
  • Supports concurrent ping-based reachability verification of unreachable VMs
  • Singleton historical_cmdb instance loaded at module import
  • Current scale: ~1,943 total records (1,795 active, ~148 decommissioned)

vcenter_health.py – vCenter health monitoring

Implements VCenterHealthChecker. Performs three independent checks per vCenter:

  • Ping: ICMP reachability + latency via subprocess.run(['ping', ...])
  • HTTPS: Web UI availability check via https://<vcenter>/ui/
  • SDK: pyVmomi SmartConnect test (validates credentials and API)

Results are persisted to ~/.vmware-dashboard-cache/vcenter_health.json. Uses ThreadPoolExecutor for concurrent multi-vCenter checks. Auto-refreshes every 300 seconds via the health loop in main.py.

puppet_client.py – PuppetDB integration

Implements PuppetClient, a REST API client for PuppetDB v4. Enriches CMDB records with:

  • OS details (name, family, release, kernel version)
  • Network facts (IP, netmask, MAC address)
  • Puppet role, environment, FQDN
  • Agent version, last report status, uptime
  • Memory facts

Endpoint: https://your-puppetdb-host:8081/pdb/query/v4
SSL verification is disabled (internal PKI). Deactivated/expired nodes are treated as not found.

global_search.py – Cross-resource search engine

Implements GlobalSearch — a lightweight, in-memory search across all resource types. Instantiated per-request with the current cache snapshot. Searches across:

  • VMs – from CMDB (vmName, ipAddress, hostName, cluster, guestOS, uuid)
  • Hosts – from live cache (hostName, clusterName, vcenterName, cpuModel)
  • Datastores – from live cache (datastoreName, vcenterName, type)
  • Networks – from live cache (networkName, vcenterName, type)

4.2 Cache & Data Refresh

PersistentCache

PropertyValue
TTL1,800 seconds (30 minutes)
Cache file~/.vmware-dashboard-cache/cache.json
Thread safetythreading.RLock on all read/write operations
PersistenceAtomic write via temp file + os.replace()
Keys storedvcenters, vms, hosts, datastores, networks, snapshots

Background Refresh Flow

  1. Triggered at startup and after any vCenter connect/disconnect event
  2. Loads all enabled credentials from vcenter_credentials.json
  3. Calls collect_vcenter_data_parallel() with up to 4 worker threads
  4. Assigns sequential IDs to all returned objects
  5. Writes results to cache: vcenters, vms, hosts, datastores, networks
  6. Collects snapshots via pyVmomi sessions
  7. Triggers CMDB sync from refreshed VM list
  8. Refresh lock prevents concurrent refresh runs

4.3 Credential Management

vCenter credentials are stored in ~/.vmware-dashboard-cache/vcenter_credentials.json. Each entry includes:

{
  "hostname": "vcenter.example.com",
  "id": "vc-12345678",
  "username": "service-account@domain",
  "password": "<encrypted at rest by OS file permissions>",
  "port": 443,
  "enabled": true,
  "lastConnected": "2026-04-25T10:30:00.000Z"
}
Security Note
Credentials are stored in plaintext JSON on disk, protected by Linux file system permissions. The file should be owned by the service account with chmod 600. Consider migrating to a secrets manager (e.g., HashiCorp Vault) for production hardening.

7. Data Flow

Initial Load (Backend Startup)

  1. vm-api.service starts → main.py startup handler executes
  2. Load all credentials from vcenter_credentials.json
  3. Connect pyVmomi sessions for each enabled vCenter
  4. Start background_refresh() in a daemon thread
  5. Seed VCenterHealthChecker with known hostnames
  6. Start async health loop (runs every 300 s)

Background Refresh (every 30 min or on-demand)

  1. collect_vcenter_data_parallel() calls each vCenter using FastVCenterCollector
  2. PropertyCollector retrieves VMs, hosts, datastores, networks in batch
  3. Results merged from all vCenters, sequential IDs assigned
  4. Written to PersistentCache → persisted to cache.json
  5. Snapshots collected via pyVmomi per-session
  6. historical_cmdb.update_from_refresh() called to sync CMDB

Page Request (Browser → API)

  1. Browser requests /datastores
  2. React Router renders DatastoresPage
  3. Component calls fetch("/api/datastores") (proxied by Vite to :8000)
  4. FastAPI reads from PersistentCache (in-memory, <1ms)
  5. Returns JSON response: {"success": true, "data": [...], "count": N}
  6. Component renders table with data

CMDB Lifecycle

  1. Every refresh, update_from_refresh(vms, connected_vcenters) is called
  2. New VMs are added with status: "active" and first_seen timestamp
  3. Existing active VMs update their last_seen and attributes
  4. VMs missing from a connected vCenter are marked status: "decommissioned"
  5. VMs from disconnected vCenters retain their previous status
  6. CMDB is persisted atomically to cmdb_history.json

12. Troubleshooting Guide

Common Issues

Dashboard shows no data / blank tables

  1. Check backend is running: sudo systemctl status vm-api
  2. Check backend logs for connection errors: sudo journalctl -u vm-api -n 50
  3. Verify at least one vCenter is connected: GET /api/health → check vcenters_connected
  4. If vcenters_connected: 0 — go to Settings and reconnect vCenters
  5. If connected but no data — trigger manual refresh: Settings → "Refresh Data"

vCenter shows "disconnected" in Settings

  1. Network issue — verify server can reach vCenter: ping <vcenter-hostname>
  2. Credential issue — click "Reconnect" and re-enter username/password
  3. Test credentials via Settings → vCenter card → "Test Connection"
  4. Check vCenter is responding: curl -k https://<vcenter>/ui/

Cache is stale / data is out of date

  1. Check cache age: GET /api/cache/status → review age_seconds per key
  2. If refresh_in_progress: true, wait for refresh to complete
  3. Force refresh: POST /api/cache/refresh or Settings → "Refresh Data"
  4. If refresh fails, check backend logs for Python exceptions

Search returns no results

  1. Ensure query is at least 2 characters
  2. Data must be in cache — trigger a refresh first
  3. VM searches use CMDB data; ensure cmdb_history.json is populated
  4. Check GET /api/search?q=test directly to isolate API vs. UI issue

vCenter Health page shows all "Unknown"

  1. Click "Check All" to trigger a fresh health check
  2. Health checks run in background — wait 30–60 seconds and refresh
  3. If still failing, check that the server can reach vCenter hosts on ICMP (ping) and HTTPS (443)

Frontend page is blank (white screen)

  1. Check browser developer console for JavaScript errors
  2. The top-level ErrorBoundary in App.tsx catches crashes — look for "Something went wrong" message
  3. Try "Try again" button to reset the error boundary
  4. Check that vm-frontend.service is running and serving the latest build
  5. Hard-refresh browser (Ctrl+Shift+R) to clear cached JS

PuppetDB enrichment shows no data

  1. Verify network connectivity to PuppetDB: curl -k https://your-puppetdb-host:8081/pdb/query/v4/nodes/ | head -20
  2. PuppetDB only has records for managed hosts — VMs without Puppet agents will show no data
  3. Check hostname matches the Puppet certname (usually FQDN)
